# HL7 ORU PDF Validation Tool

## Purpose

This note records the validation workflow for confirming that a PDF report
embedded in an HL7 v2.5.1 result message can be transmitted correctly from the ECG
application platform (AP) to a hospital-side Mirth Connect or Open Integration
Engine (OIE) endpoint.

The validation tool is intended to:

- Extract the embedded PDF from an exported HL7 message.
- Decode the Base64 payload.
- Detect transmission or export corruption.
- Write the recovered PDF to disk.
- Open the generated PDF for inspection.
- Report clear PASS, FAIL, or WARNING results.

## Environment

| Item | Value |
| --- | --- |
| OS | Ubuntu Desktop |
| Language | Python 3 |
| Message type | HL7 v2.5.1 `ORU^R01` or `ORU^W01` |
| Export source | Mirth Connect / Open Integration Engine |

## Initial Assumption

The first design assumed the input was a normal XML document with a Base64 PDF
payload directly inside an XML tag:

```xml
<Document>
    <PDF>
        JVBERi0xLjQK...
    </PDF>
</Document>
```

That design searched XML tags for large Base64 strings.

## Actual File Structure

Inspection of the real export, `message_3.xml`, showed that the XML file is
only a wrapper generated by Mirth/OIE. The actual HL7 message is stored inside a
`<content>` element:

```xml
<message>
    ...
    <content>
        MSH|...
        PID|...
        OBR|...
        OBX|...
    </content>
</message>
```

The extraction strategy therefore changed from XML tag scanning to:

```text
XML wrapper
        |
        v
Extract <content>
        |
        v
Parse HL7 message text
        |
        v
Find OBX ED segment
        |
        v
Decode embedded PDF payload
```

## PDF Location

The PDF report is carried in an `OBX` segment:

```hl7
OBX|1|ED|11504-8^ECG PDF Report^LN||AP_SIMULATOR^Application^PDF^Base64^JVBER...
```

Relevant fields:

| Field | Meaning |
| --- | --- |
| `OBX-2` | Data type |
| `OBX-5` | Observation value |

For the PDF attachment:

- `OBX-2` is `ED`, meaning Encapsulated Data.
- `OBX-5` contains ED components separated by `^`.
- The payload format is `SourceApplication^TypeOfData^DataSubtype^Encoding^Base64Data`.

Example:

```text
AP_SIMULATOR^Application^PDF^Base64^JVBER...
```

The Base64 PDF payload starts after the fourth caret.

## Parsing Strategy

### XML Wrapper

The XML export is parsed with `xml.etree.ElementTree`:

```python
root = ET.fromstring(text)
```

The tool then searches all elements for the `content` element and reads the HL7
message text from that node:

```python
for elem in root.iter():
    if elem.tag == "content":
        content = elem.text
```

### HL7 Message

The HL7 text is processed line by line:

```python
for line in content.splitlines():
```

Only `OBX` segments are considered:

```python
if line.startswith("OBX|"):
```

Each segment is split into fields:

```python
fields = line.split("|")
```

The tool only examines segments where:

```text
OBX-2 == ED
```

Then `OBX-5` is split into ED components:

```python
parts = obx_value.split("^")
```

## Base64 Detection

Each ED component is inspected for a PDF signature:

```python
candidate.startswith("JVBER")
```

PDF files begin with `%PDF`, and the Base64 encoding of `%PDF` starts with
`JVBER`. This makes `JVBER` a practical signature for locating Base64-encoded
PDF content inside the HL7 value.

## Validation Checks

### Base64 Decode

The payload is decoded with:

```python
pdf_bytes = base64.b64decode(payload)
```

PASS condition:

```text
No decoding exception is raised.
```

### PDF Header

The decoded bytes must begin with `%PDF`:

```python
pdf_bytes.startswith(b"%PDF")
```

PASS condition:

```text
True
```

### File Size

The decoded byte length is compared with the written output file size:

```python
len(pdf_bytes) == output_pdf.stat().st_size
```

PASS condition:

```text
The values match exactly.
```

This detects write-time truncation or file output problems.

### EOF Marker

The decoded PDF is checked for a trailing EOF marker:

```python
pdf_bytes.rstrip().endswith(b"%%EOF")
```

Result handling:

```text
PASS if present.
WARNING if missing.
```

A missing `%%EOF` marker may indicate a truncated transmission or an incomplete
PDF export.

## Future Improvements

### SHA256 Integrity Validation

The current check confirms the decoded byte count and output file size. A
stronger integrity check would calculate:

```python
hashlib.sha256(pdf_bytes).hexdigest()
```

If a trusted original hash is available, compare:

```text
Original SHA256
Decoded SHA256
```

This confirms byte-for-byte integrity.

### Multiple PDF Support

Current behavior stops after the first valid PDF payload is found. A future
version should support multiple ED PDF segments:

```text
output_1.pdf
output_2.pdf
output_3.pdf
```

### HL7 Parsing Library

The current approach uses `split("|")` and `split("^")`, which is sufficient
for the controlled validation file. A production-grade tool should consider an
HL7 parser such as `hl7apy` or `python-hl7` for proper HL7 field handling.

## Outcome

The validation workflow confirms that the simulator-generated PDF report can be
recovered from a Mirth/OIE XML export containing an HL7 v2.5.1 result message. The
tool validates the Base64 payload, PDF header, output file size, and EOF marker,
then writes a generated PDF for manual inspection.

This provides a practical lab check for PDF report transmission between
hospital systems and ECG AP implementations.
